1. Executive Summary

The renewal of the production X.509 certificate protecting the SAP NS2 OpenText Core Archive Connector on host 10.168.0.32 was completed successfully, two days ahead of the expiration deadline. The new certificate, signed by Parker Hannifin General Issuing CA v2, has been deployed to all three production keystores, the OpenText service has been restarted cleanly, and rollback artifacts have been retained.

The renewal required approximately eleven hours of continuous work rather than the ten minutes typical for a routine Java keystore cert replacement. The additional time is attributable in its entirety to missing documentation, missing cryptographic artifacts that should have been delivered with the signed certificate, and environmental network constraints that prevented use of standard chain-resolution paths.

None of the delay is the result of process error or tooling failure on the deployment side. Every hour beyond the expected envelope was spent either (a) discovering information that should have been captured in the runbook, (b) obtaining artifacts that should have been delivered by the issuing CA, or (c) working around a platform-level network restriction that blocked automated chain download.

4. What Went Wrong, Step by Step

This section documents the actual diagnostic path followed during the incident window, including every blocker encountered and how each was resolved. It is intended to serve as both an audit record and a reference for the next renewal cycle.

Phase A - Environment preparation & first import attempt Hours 0 - 1
  • Jump host session established. Administrative SMB share access to target host confirmed. Signed leaf certificate OTP.cer delivered by the issuing CA was staged at C:\temp\script\OTP.cer.
  • Java keytool located at C:\Program Files\Java\jre1.8.0_481\bin\keytool.exe.
  • First import attempt executed against the production keystore referenced in the existing runbook (E:\CERTS\s4pceotcac.p12, copied locally to avoid mutating production).
Blocker 1

keytool error: Public keys in reply and keystore don't match

This error indicates the signed certificate's public key does not correspond to the private key stored under the target alias. The signed certificate was generated against a different keystore than the one the runbook documents.

Phase B - Locating the correct source keystore Hours 1 - 4
  • Enumerated the target host's file system over SMB to identify all Java keystore files (.p12, .pfx, .jks, .pse). Search roots included E:\CERTS, E:\OTC, E:\SAP, C:\Program Files\SAP, C:\Program Files\sapdb, and user profile directories.
  • SMB recursion across user profile directories on the target host proved extremely slow over the jump host link. Shifted strategy to targeted, high-probability candidates.
  • Implemented public-key-based keystore identification using the .NET X509Certificate2Collection API: iterate each candidate keystore, extract every private-key entry, compute the SHA-256 hash of its public key, and compare against the SHA-256 hash of the public key in the signed certificate. A byte-level match positively identifies the keystore bound to the CSR.
  • Match located in E:\CERTS\s4pceotcac_new.p12 - a staging keystore created during Phase 1 of this renewal cycle, not referenced anywhere in the runbook.
  • Cross-confirmed with a keytool import against the identified keystore: keytool error: Failed to establish chain from reply - a different error class, indicating private key matches but chain validation is incomplete. This confirmed we had the correct source keystore.
Phase C - Chain resolution attempts Hours 4 - 8
  • Java keytool requires full trust-chain resolution when replacing an existing private-key entry. The signed leaf certificate was delivered in isolation, without the accompanying intermediate or root CA certificates.
  • Attempt 1 - Windows certificate store chain build: Invoked System.Security.Cryptography.X509Certificates.X509Chain to let Windows construct a chain from its local trust stores. Result: chain length = 1 (leaf only). Neither Parker intermediate nor root is present in the jump host's trust stores - they are internal Parker Hannifin authorities, not public CAs, and the jump host is not part of Parker's domain.
  • Attempt 2 - AIA automated download: Parsed the Authority Information Access (AIA) extension embedded in the signed certificate. Located the URL http://www.parker.com/crl/Parker%20Hannifin%20General%20Issuing%20CA%20v2(1).crt. Invoke-WebRequest attempt returned HTTP 403 Access Denied with an Akamai edge-server reference ID. The URL is configured to serve content only to clients originating from Parker corporate IP ranges. Requests from AWS GovCloud fail at the edge.
Blocker 2

Both standard automated chain-resolution paths (local trust store and AIA download) were blocked. A non-standard approach was required.

Phase D - Chain extraction from prior production keystore Hours 8 - 10
  • Hypothesis: the previously deployed production keystore (E:\CERTS\s4pceotcac.p12, in service since 2024) must contain the Parker chain as trusted entries; otherwise Java keytool would not have accepted its original import either.
  • Loaded the prior keystore via X509Certificate2Collection.Import with the Exportable storage flag. Iterated its entries and confirmed the presence of three certificates: leaf (CN=ns2otpapp.sap.parker.corp), intermediate (CN=Parker Hannifin General Issuing CA v2), and root (CN=Parker Hannifin Root CA v2).
  • Exported both CA entries as DER-encoded .cer files. These are exactly the artifacts that should have been delivered alongside the signed leaf certificate.
  • Imported both CAs into the working keystore under distinct aliases (parkerca0, parkerca1), followed by import of the renewed leaf under the established alias s4hpce.
  • Final keytool response: Certificate reply was installed in keystore. Chain validation succeeded end-to-end.
Phase E - Deployment, restart, verification Hour 10 - 11
  • Created timestamped backups of all three production keystore paths on the target host, suffixed .bak-20260424073045.
  • Distributed the renewed keystore to all three production paths over SMB. All three are now byte-identical.
  • Discovered the OpenText Windows service on the target host by enumerating remote services filtered on display name and service name patterns. Identified as otacc, DisplayName "Core Archive Connector - otacc", prior state Running.
  • Restarted the service via Restart-Service -ComputerName 10.168.0.32 -Name otacc -Force. Post-restart status verified: Running.

6. Engineering Decisions That Preserved Production Safety

Given the number of unknowns encountered, multiple decisions were made deliberately to avoid introducing risk to the production service. These are documented here so the record is clear that the extended duration reflects engineering discipline, not delay.

  1. No production keystore was mutated during investigation. Every diagnostic import was performed against a local copy. The production keystore remained untouched until the renewed version was verified as fully chain-valid.
  2. Positive keystore identification was required before import. Rather than iterating trial imports against production paths - which can corrupt a keystore on partial failure - public-key SHA-256 comparison was used to identify the correct source keystore deterministically before any write operation.
  3. All three production paths were backed up with a single timestamp. Rollback to the pre-renewal state is a single-step operation. Backups are suffixed .bak-20260424073045 and retained on the target host.
  4. Chain material was sourced from an already-trusted internal artifact. Rather than accepting an arbitrary certificate that claimed to be Parker's intermediate, the intermediate and root were extracted from the prior production keystore. This is the same material the service has trusted since 2024, which guarantees continuity of trust for both the Java runtime and any upstream validators.
  5. Service restart was explicitly verified post-deployment. Service status was re-queried after restart via RPC, not inferred from the restart command's exit code alone.

7. What Should Have Been Provided

For reference and for future cycles, the following artifacts and pieces of information should be delivered as part of any cert renewal handover for this application. Their absence was the direct cause of the extended duration of this renewal.

  1. Full certificate chain as a PKCS#7 bundle (.p7b) or individual .cer files. Delivered alongside the signed leaf. Must include the intermediate CA and the root CA.
  2. Identification of the source keystore used for CSR generation. Phase 1 of this cycle generated a dedicated staging keystore. The corresponding path must be captured in the handover notes so that the deployment engineer knows which file holds the matching private key.
  3. Canonical list of all keystore paths the target service reads from. The application reads its keystore from three paths on this target host. All must be documented in the renewal runbook.
  4. Target Windows service name and DisplayName. Currently undocumented; discovered via remote enumeration during this cycle.
  5. Platform network access constraints for CDP / AIA endpoints. The jump host cannot reach Parker-internal CA distribution URLs. This should be captured so that future engineers do not spend time attempting automated chain download.

8. Recommendations

  1. Update the renewal runbook to reference both keystores. Explicitly document s4pceotcac_new.p12 as the staging keystore and the path that will hold the matching private key for any in-flight CSR.
  2. Document all three synchronized keystore paths including the OpenText runtime path, and update deployment tooling to push to all three atomically.
  3. Formalize the intermediate-chain request at CSR-signing time. Ask the Parker PKI team to return signed certificates as a PKCS#7 bundle including the full chain. Alternatively, distribute the intermediate and root CAs to the renewal engineer's toolkit as long-lived reference artifacts (they rotate on a multi-year cadence).
  4. Record the target Windows service name (otacc) and the hosting platform details (AWS GovCloud us-gov-west-1a, instance class) in the application inventory record.
  5. Retain the scripts and PowerShell one-liners produced during this incident as the canonical renewal tool. These artifacts now handle keystore identification, chain extraction from the prior keystore, import, distribution, backup, and service restart in a single deterministic sequence. Next year's renewal should complete within the originally expected ten-minute window.
  6. Calendar the next renewal for 2027-03-24. One-month buffer ahead of the 2027-04-24 expiration.
