HOW TO RUN THE SCRIPT ON NS2SW1APP1

Follow these steps exactly. RDP into the server, copy the script, and run as Administrator.

1
RDP Into the Server

Connect via RDP to ns2sw1app1 at IP 10.134.4.171 using your admin credentials.

2
Copy the Script to the Server

Copy Fix-RDP-Cert-NS2SW1.ps1 to the following folder on the server. Create the folder if it does not exist.

C:\Certs\Fix-RDP-Cert-NS2SW1.ps1
3
Open PowerShell as Administrator

Click Start, search for PowerShell, right-click it and select Run as Administrator. This is required — the script will not work without admin rights.

4
Allow Script Execution and Navigate to Folder

Paste these two commands into PowerShell and press Enter after each one:

Set-ExecutionPolicy Bypass -Scope Process -Force cd C:\Certs
5
Command 1 - VALIDATE - Check Current Certificate

Run this first to see the current state of the certificate, key size, IIS binding, and RDP binding.

.\Fix-RDP-Cert-NS2SW1.ps1 -Step VALIDATE

What it checks:

  • Existing cert-sw1 in the machine certificate store
  • Key size — must be 2048-bit
  • Private key presence
  • SAN (Subject Alternative Name) — DNS and IP entries
  • Certificate expiry
  • IIS HTTPS binding on port 443
  • RDP certificate binding via WMI
All output and logs are automatically saved to C:\Certs\cert-fix-log.txt
6
Command 2 - GENERATECSR - Create New 2048-bit Certificate Request

This generates a brand new private key and CSR on the server with all correct parameters.

.\Fix-RDP-Cert-NS2SW1.ps1 -Step GENERATECSR

What it creates:

  • A new 2048-bit private key stored securely on ns2sw1app1
  • CSR file at C:\Certs\ns2sw1app1-CSR.txt
  • CN set to ns2sw1app1.ns2corp.local
  • SAN includes DNS: ns2sw1app1, DNS: ns2sw1app1.ns2corp.local, IP: 10.134.4.171
After running this command: Send the file C:\Certs\ns2sw1app1-CSR.txt to the Certificate Authority for signing. Wait for the signed certificate file to be returned before running Step 7.
7
Command 3 - IMPORT - Install Signed Certificate and Bind Everything

Run this once the CA returns the signed certificate file. Replace signed.cer with the actual filename the CA sends back.

.\Fix-RDP-Cert-NS2SW1.ps1 -Step IMPORT -SignedCertPath "C:\Certs\signed.cer"

What it does automatically — no manual steps needed:

  1. Runs certreq -accept to import and pair the signed cert with the private key on ns2sw1app1
  2. Sets the certificate friendly name to cert-sw1
  3. Removes any old HTTPS binding from IIS
  4. Creates a new IIS HTTPS binding on port 443 with the new certificate
  5. Falls back to netsh if WebAdministration module is unavailable
  6. Binds the certificate to RDP services via WMI
  7. Falls back to registry binding if WMI is unavailable
  8. Restarts the RDP service (TermService) to apply all changes
  9. Runs final validation and prints PASS or WARN for each item
  10. Saves full log to C:\Certs\cert-fix-log.txt
After this runs successfully the RDP certificate error will be resolved and users will be able to connect without invalid certificate warnings.

HOW THE CERTIFICATE PROCESS WORKS

This section explains the standard certificate workflow so all parties have a clear shared understanding of each step.

StepResponsible PartyAction
1IT InfrastructureGenerate CSR on ns2sw1app1. Private key is created and stays on the server. Only the CSR file is shared.
2IT InfrastructureSend the CSR file to the Certificate Authority for signing. This is the only file that leaves the server.
3Certificate AuthorityCA signs the CSR and produces a signed certificate (.cer or .crt). The signed certificate is returned to IT Infrastructure.
4IT InfrastructureImport the signed cert on ns2sw1app1. Script pairs it with the private key and binds to IIS and RDP automatically.
The certificate is issued for ns2sw1app1.ns2corp.local (IP: 10.134.4.171). The signed certificate must be returned to IT Infrastructure for installation on that server, as the matching private key only exists there.

CERTIFICATE FILE TYPES EXPLAINED

File TypeContainsPurpose
CSR (.txt)Public key request onlySent to CA for signing. Does not contain private key.
.cer or .crtSigned certificate onlyReturned by CA to IT Infrastructure. This is what gets installed on ns2sw1app1.
PFX / PKCS12Cert AND private key with passwordUsed for backup only. Created by IT Infrastructure after installation. Not required by CA.
Private KeyPrivate key onlyRemains on ns2sw1app1 at all times. Never shared externally.

Does the Script Create a PFX

Yes. After the signed certificate is imported, you can optionally export a PFX backup file using this command:

certutil -exportPFX -p "YourPassword" My cert-sw1 C:\Certs\cert-sw1-backup.pfx

This creates a password-protected PFX for backup. This step is optional and performed after the certificate is fully operational.
